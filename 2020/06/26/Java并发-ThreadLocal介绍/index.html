Java并发-ThreadLocal介绍

2020-06-26

ThreadLocal是JDK提供的、基于“线程封闭”实现的保证线程安全的策略。顾名思义，它提供线程本地变量，如果创建一个ThreadLocal变量，那么访问这个变量的每个线程都会有这个变量的一个副本，所以在并发情况下，每个线程操作的是自己本地内存中的变量，从而规避了线程安全问题。

本博客将从应用场景到实际原理介绍Java并发中的「ThreadLocal」。

看似问题圆满解决了，不过仔细分析：在上面的代码中，用了10个线程执行1000个任务，每个任务中都调用myFormatDate方法来获取指定时间戳对应的日期，而在myFormatDate方法内部，会new一个SimpleDateFormat对象，用来转化时间格式。也就是说，转化时间格式的任务共有1000个，也就调用了1000次myFormatDate方法，也就会在堆上1000次创建SimpleDateFormat对象，而每个任务需要的SimpleDateFormat对象都是一致的，为每个任务创建一个单独的SimpleDateFormat对象这对内存是一种浪费，虽然说在随着myFormatDate执行完毕，下次jvm垃圾回收的时候会回收掉在堆上分配的垃圾对象，但是我们还是应该尽可能避免这种写法以免垃圾回收频繁发生。

尝试的改进思路：我们可以把这个通用的SimpleDateFormat对象作为类的成员变量。这样一来，每次调用myFormatDate方法转化日期，都是共用一个SimpleDateFormat对象，在堆内存中也只需要为一个SimpleDateFormat对象分配内存空间。

如下图，根据此思路改进后，执行，会发现输出结果有误——有部分重复的日期输出，显然和代码想表达的含义不符。

究其原因，其实是因为SimpleDateFormat类不是线程安全的，它其中保存了一些时间相关的状态变量，对这些变量的并发修改不是线程安全的，所以导致输出的日期有部分重复，导致错误。

真正的救星—ThreadLocal：在提出解决方案之前，让我们再次梳理一下现在的困境：

在第一种解决方案中，SimpleDateFormat对象以局部变量在方法内部，其它线程无法访问，自然没有线程安全问题。

而在第二种方案中，SimpleDateFormat对象作为类的成员变量，线程之间共享访问，暴露出了线程安全问题。

第一种解决方案为1000个任务创建1000个SimpleDateFormat对象能得到正确结果，但是太浪费内存空间了！

而第二种解决方案1000个任务共用1个SimpleDateFormat对象，节约了内存，但是有线程安全问题，不能保证得到正确结果！

为此我们就想，有没有一种折中的解决方法 —— 让每个线程内部保存一个SimpleDateFormat对象，各个线程之间保存的是不同的SimpleDateFormat对象，这样自然也就不存在线程安全问题了，同时还节约了堆内存空间。这就引出来我们本文的主角ThreadLocal了。
